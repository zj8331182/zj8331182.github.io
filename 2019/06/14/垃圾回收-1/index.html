前言

  • 谈到Java语言，垃圾回收必然是一个绕不过去的题，那么垃圾回收是什么东西，用我自己话讲其实和现实中扔垃圾差不多，无非就是把没用的东西清理掉，那就从下面几个角度来了解一下，Java是如何“扔垃圾”的

  • 在Android中可能会有一些不同。我们都知道在5.0之前Android使用的是Dalvik虚拟机，5.0之后是ART，这两者在GC上有一定差异，我们后续的所有介绍，会区别两者来介绍

回收的时机：什么时候清理

  1. 内存到达伐值

    生活中如果我们新买了一个s东西，但是拿回家后发现没地方放了，这时就需要清理没用的垃圾来腾出地方给新的东西，在JVM中同样，当新建对象导致内存到达伐值的时候就会触发GC。

  2. 显示调用

    有的时候可能家里突然来了客人，或者某天你心情好，突然就想要收拾一下屋子，这个时候你就自发的清理了一下垃圾。在JVM中你可以调用‘System.GC.Collect();’来手动清理。（当然，这个一般是不建议手动调用的）

  3. 出现一些突发情况

    突然有一天你被告知继承了七舅老爷的亿万遗产，要搬到独栋大别墅里，临走之前一定要整理一下，这种情况在JVM中也有存在，当然比喻的可能不恰当。

  4. Android中的时机

    • 提到这里就必须要提到四种GC了，他们分别是：

      • GC_FOR_MALLOC: 表示是在堆上分配对象时内存不足触发的GC。

      • GC_CONCURRENT: 当我们应用程序的堆内存达到一定量，或者可以理解为快要满的时候，系统会自动触发GC操作来释放内存。

回收对象： 如何识别垃圾

  1. 什么样的对象为垃圾

    在JVM的世界里，如果一个对象是存活的，他都会被其他对象引用，如果没有对象引用他，那么就说明他没有被使用，也就说明他已经死亡，是需要被清理的垃圾。

    如何找到垃圾，这个问题听上去很简单，按照上边说的只要找到没有被其他对象引用的对象就可以，那么该如何找呢？有以下几个方法。

  2. 引用计数

    引用计数的实现非常简单，就是每一个对象都有一个计数器，当一个对象被另一个对象引用时，计数器加一，当引用失效时计数器减一，当计数器为0时则说明该对象没有被其他对象引用。

    引用计数法虽然看上去实现简单，但是有一个缺点，就是当两个对象相互引用，而没有其他对象引用他们时，这两个对象其实已经是无用对象了，但是他们的计数器不为0，所以永远不会被回收

    该方式非JAVA采用的垃圾回收方式

  3. 可达性分析

    由于上文说到的引用计数有其弊端，JVM采用了另外一种方式来标记对象是否存活，具体的方式如下

    这里的GC ROOT指的是一下几种对象

    • JVM栈(栈帧数据中的本地变量表)中引用的对象。

    • 方法区中常量引用的对象。

    需要注意一点，即使在可达性分析算法中不可达对象，也并非是“非死不可”的，要真正宣告一个对象的死亡，至少需要经历两次标记的过程：

    如果一个对象在进行可达性分析之后发现没有与GC Roots相连的引用链，那么他将会第一次标记并且进行一次筛选，塞选的条件是是否复写finalize（）。当对象没有复写finalize()方法，或者finalize()方法已经被虚拟机调用过，虚拟机讲着两种情况都视为“没有必要执行finalize()方法”。

    如果这个对象被判定为有必要执行finalize()方法，那么这个对象会被加入一个“F-Queue”队列中，并在稍后由一个虚拟机建立的、优先级低的Finalize线程，去触发这个方法，但并不承诺会等待他运行结束。（非阻塞）

    finalize()方法是对象逃脱死亡厄运的最后一次机会，稍后的GC会对在“F-Queue”队列中的对象进行第二次小规模的标记；

    如果对象要在finalize()中拯救自己，只需要重新与引用链上的对象就行关联即可，那么在第二次标记时它将被移出“即将回收”的集合；
      
    如果对象这个时候还是没有逃脱，那基本上他就真的被回收了。

  4. 方法区的回收

    上面说到的可达性分析是用于对存储在堆中的对象，那么对于相对来说GC频率比较低的方法区主要回收的对象有以下两个方面

    • 废弃常量

      该部分和上文中的Java堆的回收比较类似，就是回收没有被引用的常量，比如一个叫“abc”的字符串常量，如果没有任何一个String叫“abc”，则该常量就会被认定为废弃常量

    • 无用的类

      首先先要区别一下，类和对象，对象是类的实现，我们可以理解是对象是new出来的类，而类是我们写代码时的.class的对象。

      要认定一个类是无用的需要以下三个条件，缺一不可，相对于对象和常量来说，更为苛刻

      • 该类的所有实例都已经被回收，也就是说Java堆中无该类的对象

      • 该类的ClassLoader已经被回收

      • 该类的.class对象没有被引用，任何地方没有通过反射访问该类

回收的方式

  1. 几种常见的回收算法

    • 标记-清除

      如图，该算法的实现就是先遍历一遍所有对象，标记出需要回收的对象，再将所有需要回收的对象回收。

      该算法的缺点是碎片化严重，当需要申请连续的大内存块时，明明有足够的空间，却因为不连续导致申请失败，进而触发多次GC

    • 复制

      如图，该算法是将区域分成两份，在区域1遍历，将不需回收的对象则依次复制到区域2里面，等到全部遍历完成之后将区域1全部清除

      该算法的缺点同样明显，就是只有一半的空间被真正使用，并且如果一个对象一直存活的话，没回回收都需要进行一次复制，效率低下

    • 标记-整理（标记-压缩）

      如图，该算法是遍历所有内存，将不需回收的对象向前移动直到上一个不需回收的对象的尾部，最后将除不需回收的对象的部分全部回收

      该算法的缺点就是效率比较低，当大多数的对象是需要回收的时候，会进行多次移动对象的操作

    • 分代收集

      上面的三种算法各自有各自的优劣，结合生活中，如果一个东西你每回收拾垃圾都把它保留下来了，那说明这个东西对你比较重要，它被清理的可能性也就比较小，那我们就可以把这一类东西放到一块，下回收拾的时候，就先不收拾这块，等到是在没地方再收拾它们。这就是分带整理，如下图。

      如上图JVM Heap被分为两大部分（Young Gen 和 Old Memory）

      其中Old Memory就是上边所说的“每回收拾垃圾保留下来的东西”这部分对象中大部分是不需回收的对象，所以这一部分使用的算法是标记-整理算法，从而保证空间的利用率

      而Young Gen则是采用了比较特殊的复制算法来实现，这里将Young Gen分为三个区域，其中最大的叫Eden，新对象创建时都是在这个区域，另外两个叫Survivor，Eden：Survivor的比例一般为8:1。GC时，将Eden和其中一个Survivor中的对象使用复制的算法统一复制到另外的一个Survivor中，这个时候呢如果有很多的对象都不能回收，就可能会出现另一个Survivor不够放的情况，这时就会启用Old Memory来存放。

      为什么Young Gen要采用这种方式呢，原因是，新创建的对象的生命周期普遍较短，也就是挺不多一次GC，说有当有大量对象需要回收时，采用复制算法，它的第二个缺点“对象长时间存活，需要多次复制”的缺点就相应减少，而采用Eden和Survivor这种内存分配模式，有极大的提高了内存利用率，这样复制算法的第一个缺点也相应消除，所以，对于新生代这种生命周期相对短暂的对象，复制算法是最理想的方法

    • Android中的GC

      上面说到的是常规JVM中采用的模式，当然这里面没有深入介绍8大垃圾回收器，下面会介绍Dalvik和ART中的垃圾回收策略

      • Dalvik

        提到Dalvik的垃圾回收就要说到CMS(Concurrent Mark Sweep),看到Mark Sweep大家应该也猜到了是标记-清除的算法，这里的标记-清除和我们上文介绍的不太一样，不一样的地方有，下图描述了Dalvik的GC过程

        1. 标记分为两步

          第一步先标记GC_ROOT，这个一步是需要Stop the world，也就是需要停掉除GC线程外的说有线程，以防止引用关系发生变化。

          第二步标记其他的对象，这个阶段是允许其他线程运行的，但是如果其他线程在这段时间使引用关系发生了变化，需要记录下来这些变化，在第二步标记完成后，需要再对记录下来的这些对象进行一次标记，这个过程是需要Stop the world 的，但是由于这部分的对象会比较少，所以不会stop太长时间

        2. 标记方式

          这里的部分清理指的是Dalvik中使用两个Bitmap来记录说有对象的状态一个，一个称为Live Bitmap，另一个称为Mark Bitmap。Live Bitmap用来标记上一次GC时被引用的对象，也就是没有被回收的对象，而Mark Bitmap用来标记当前GC有被引用的对象。有了这两个信息之后，我们就可以很容易地知道哪些对象是需要被回收的，即在Live Bitmap在标记为1，但是在Mark Bitmap中标记为0的对象。

        通过上面对JVM的GC算法的介绍我们可以知道，Dalvik的GC存在内存碎片话的问题.

      • ART

        ART的GC相对比较复杂，可能要单独一篇文章才能讲的清，我们在这就先介绍一下ART的优点，之后再写一篇单独的ART的GC的文章

        1. ART运行时堆的划分和管理更细致，它分为Image Space、Zygote Space、Allocation Space和Large Object Space四个Space，再加上一个Allocation Stack。其中，Allocation Space和Large Object Space和Dalvik虚拟机的Zygote堆和Active堆作用是一样的，而其余的Space则有特别的作用，例如，Image Space的对象是永远不需要回收的。

        2. ART运行时的每一个Space都有不同的回收策略，ART运行时根据这个特性提供了Mark Sweep、Partial Mark Sweep和Sticky Mark Sweep等三种回收力度不同的垃圾收集器。其中，Mark Sweep的垃圾回收力度最大，它会同时回收Zygote Space、Allocation Space和Large Object Space的垃圾，Partial Mark Sweep的垃圾回收力度居中，它只会同时回收Allocation Space和Large Object Space的垃圾，而Sticky Mark Sweep的垃圾回收力度最小，它只会回收Allocation Stack的垃圾，即上次GC以后分配出来的又不再使用了的对象。力度越大的垃圾收集器，回收垃圾时需要的时候也就越长。这样我们就可以在应用程序运行的过程中根据不同的情景使用不同的垃圾收集器，那就可以更有效地执行垃圾回收过程。

        3. ART运行时充分地利用了设备的CPU多核特性，在并行GC的执行过程中，将每一个并发阶段的工作划分成多个子任务，然后提交给一个线程池执行，这样就可以更高效率地完成整个GC过程，避免长时间对应用程序造成停顿。

小知识点

  • 四种引用方式

    • 强引用（Strong Reference）：程序代码中普遍存在的，普通对象创建时默认是强引用关系，只要强引用还存在，垃圾收集器就不会收集这些对象。

    • 软引用（Soft Reference）：用来描述一些还有用但非必需的对象，如果一个对象还存在着软引用关联，则虚拟机在发生OOM异常前，会将这些软引用对象进行二次回收。实现软引用的方式：SoftReference类。

    • 弱引用（Weak Reference）：也是用来描述非必需对象，但强度较软引用更弱，在下一次GC时就会被回收。实现弱引用方式：WeakReference类。

    • 虚引用（Phantom Reference）：虚引用也称幽灵应用或者幻影引用，是最弱的一种引用关系，一个对象是否是虚引用不会对其生存时间造成印象，也无法通过虚引用来获取对象实例。设置虚引用的唯一目的就是在该对象被GC时能够收到一个系统通知。
